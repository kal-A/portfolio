# Voice and Positioning

## Overall goal

Make the portfolio feel like a polished, artifact-backed product/design/systems
portfolio.

It should help Kamal apply for:

- Product Manager roles
- Technical Product Manager roles
- Associate Product Manager roles
- Product Design roles
- UX/UI roles

The portfolio should show range across:

- Product thinking
- UX research
- UX/UI design
- Visual design
- Systems thinking
- Operations/process design
- Technical implementation
- Documentation and handoff
- Cross-functional execution

The site should not feel like a generic AI-generated portfolio or a bare-bones
SaaS landing page. It should feel like a real person's project archive:
polished, thoughtful, visual where artifacts exist, and honest where they do
not.

## Voice

Write in a voice that feels like Kamal:

- Direct
- Grounded
- Thoughtful
- Systems-minded
- Practical
- Honest about constraints
- Comfortable with messy real-world workflows
- Slightly conversational, but still professional
- Clear without sounding corporate
- Confident without exaggerating

### Avoid generic AI/corporate language

- "I build products end to end"
- "0→1"
- "user-centric"
- "seamless"
- "leveraged"
- "transformed"
- "revolutionized"
- "innovative solutions"
- "cross-functional collaboration"
- "shipped impact"
- "at the intersection of..."

### Prefer concrete language, e.g.

- "I like working on messy workflows where user needs, operations, design,
  and technical constraints all collide."
- "I make unclear systems easier to understand, use, and improve."
- "I usually start by mapping the workflow before touching the interface."
- "The interface matters, but so do the handoffs, edge cases, and constraints
  behind it."

## Core positioning

Position Kamal as: **a systems-minded product/design person.**

Not just a PM. Not just a TPM. Not just a UX designer. Not just a developer.

The portfolio should show range, but it should not become vague.

Suggested positioning lines (pick or adapt one, don't use all three):

> I design products for messy real-world workflows.

> I make unclear workflows easier to use.

> I work across product, UX, operations, and technical systems — mapping the
> messy parts first, then turning them into clearer flows, prototypes, tools,
> and handoffs.

Use "University of Waterloo," not just "Waterloo."
